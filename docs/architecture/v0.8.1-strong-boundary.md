# VPSKit v0.8.1 Strong Architecture Boundary

## Definition

A hard boundary means experimental code is not just separated by convention.
It is actively blocked from production runtime, guarded by CLI filtering, and
checked by repository validation scripts.

## Forbidden In Core

The core layer must not reference or import:

- `vpskit.experimental.*`
- order processing logic
- payment logic
- automation delivery logic
- bot execution paths

## Enforcement Mechanisms

The boundary is enforced by three layers:

1. CLI command filtering for experimental verbs.
2. Guardrail checks that scan the repository for forbidden references.
3. Dedicated quarantine paths for experimental modules.

## Why Soft Isolation Was Not Enough

Soft isolation depends on convention. That is fragile in a production codebase
because one import, one command dispatch change, or one shared helper can make
experimental behavior reachable again.

v0.8.1 hardens the boundary so accidental reuse fails fast.

## Experimental Module Rules

- Experimental modules are disabled by default.
- Core modules must never import experimental modules.
- Experimental code must remain stub-only unless explicitly reviewed.
- Any new experimental surface must be quarantined outside production paths.

## Production Rule

No SaaS logic in the core repository.

Core remains limited to CLI, subscription export, bundle, demo, QA, verify,
and protocol configuration management.
