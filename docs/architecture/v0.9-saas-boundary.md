# VPSKit v0.9 SaaS Architecture Split

## Purpose

v0.9 formalizes a repository split between the stable core infrastructure
system, the commercial SaaS layer, and the client-facing UI layer.

The goal is separation of concerns, not feature expansion.

## Layers

### CORE

Stable, stateless infrastructure code.

Allowed:

- CLI core
- QA and verify
- subscription exports
- bundle generation
- demo packaging
- protocol configuration data

### SAAS

Commercial and customer-facing service layer.

Allowed:

- API boundary code
- billing orchestration
- order handling
- telegram bot integration
- webhook handlers

SAAS must not be imported by core runtime code.

### CLIENT

Presentation-only surface.

Allowed:

- landing pages
- YouTube traffic pages
- documentation site assets

Client code must not implement protocol, billing, or order logic.

## API Boundary

Core exports:

- subscription bundles
- QA reports
- protocol configs

SAAS consumes:

- bundles
- subscription files
- redacted QA artifacts

SAAS produces:

- external order records
- payment state
- delivery notifications

Core must not depend on those outputs.

## Separation Rules

- No SaaS logic in the core repo.
- No import path from core into SaaS.
- No business logic in client pages.
- No runtime coupling between the stable CLI and commercial services.

## Why This Split Exists

The repository needs a clear product boundary:

- core stays reliable and testable
- SaaS evolves independently
- client assets stay UI-only

That separation prevents commercial workflows from leaking into the stable
infrastructure toolchain.
