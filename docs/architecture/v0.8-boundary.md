# VPSKit v0.8 Architecture Boundary

## Purpose

VPSKit v0.8 formalizes a clean separation between the stable product core and any
experimental automation surfaces. The goal is to prevent architecture drift and
keep the repository safe for operational use.

## Core Layer

The core layer is the stable, production-facing system.

Allowed in core:

- VPN protocol logic
- CLI dispatch and stable utilities
- QA and verify logic
- subscription export
- bundle and demo generation

Core must remain free of SaaS, payment, and order-processing logic.

## Experimental Layer

The experimental layer is quarantined and disabled by default.

Allowed in experimental:

- bot stubs
- order stubs
- automation placeholders

Experimental modules must be stub-only unless explicitly approved in a future
review. They must not be imported by the stable CLI, QA, verify, subscription,
bundle, or demo flows.

## Hard Rule

No SaaS logic in the core repository.

This means:

- no payment processing
- no order fulfillment automation
- no customer lifecycle backend
- no hidden delivery side effects in stable commands

## Why This Exists

The boundary exists to prevent the repository from drifting from a stable VPN
toolkit into an automation-revenue system. The stable path stays predictable and
testable, while experimental ideas remain isolated, documented, and easy to
remove or replace.
